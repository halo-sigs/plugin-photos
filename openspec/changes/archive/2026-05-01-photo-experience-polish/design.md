## Context

图库控制台存在多个已知体验问题：客户端搜索与服务端分页冲突、照片无法跨分组移动、批量操作能力弱、照片排序不可控、空状态引导不足。

## Goals / Non-Goals

**Goals:**
- 修复搜索逻辑，统一为服务端搜索
- 支持单张照片和批量跨分组移动
- 支持照片拖拽排序
- 完善空状态引导和错误提示

**Non-Goals:**
- 相册封面自动选择算法
- 图片压缩或格式转换
- 回收站/软删除机制（已有 deletionTimestamp）

## Decisions

**1. 移除 Fuse.js，完全使用服务端搜索**
- 选择：删除前端 Fuse.js 依赖，keyword 只通过 API params 传递
- 理由：客户端搜索在分页场景下结果不完整，与分页状态冲突；服务端搜索支持全文索引
- 替代方案：保留 Fuse 但禁用分页（不适合大量照片场景）

**2. 批量操作使用现有单资源 API 并发调用**
- 选择：前端并发调用 PUT/DELETE 单资源接口
- 理由：无需新增后端批量接口，实现简单；Halo Extension API 本身适合单资源操作
- 替代方案：后端新增批量接口（减少请求数但增加后端复杂度）

**3. 照片拖拽排序复用 VueDraggable + priority 更新**
- 选择：与 GroupList 相同的模式，拖拽后批量更新 priority
- 理由：与现有分组排序模式一致，用户学习成本低
- 替代方案：独立排序字段或前端只维护顺序（不持久化）

**4. 选中状态分离：selectedPhoto vs selectedPhotos**
- 选择：selectedPhoto 用于编辑弹窗导航（蓝色高亮），selectedPhotos 用于批量操作（checkbox 勾选）
- 理由：当前 isChecked 混用两者导致编辑时高亮环跳动
- 替代方案：单选时自动清空批量选择（改变现有行为）

## Risks / Trade-offs

- **[Risk]** 批量操作大量照片时并发请求过多 → **Mitigation**：限制并发数（如每次 5 个），或提供进度提示
- **[Risk]** 拖拽排序在分页场景下受限（只能排序当前页） → **Mitigation**：列表模式支持更大分页；拖拽排序在完整列表中体验更好
